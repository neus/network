---
title: Signing Format
description: Why verification requests require a wallet signature and how to sign the deterministic message.
---

Verification requests require a wallet signature to prove ownership. The server returns a deterministic string—you sign it, we verify.

## Why Sign?

Signing binds the verification request to your wallet. It proves you control the address and authorizes the proof creation. Without a valid signature, the API rejects the request.

## Production Path

Do not hand-build the message. Always use the standardize endpoint.

<Steps>
  <Step title="Standardize request">
    Call `POST /api/v1/verification/standardize`
  </Step>
  <Step title="Sign the string">
    Sign the returned `signerString`
  </Step>
  <Step title="Submit verification">
    Submit the same normalized request body to `POST /api/v1/verification`
  </Step>
</Steps>

## Six-Line Format

The server returns a deterministic six-line string from `POST /api/v1/verification/standardize`. Sign that exact string:

```text
NEUS Verification Request
Wallet: <address>
Chain: <chainId or CAIP-2 chain>
Verifiers: <comma-separated-ids>
Data: <standardized-json>
Timestamp: <unix-ms>
```

Do not hand-author this message. Always use the standardize endpoint.

## SDK Path

Use `standardizeVerificationRequest(...)` from `@neus/sdk` to get the signing string. The SDK keeps your client aligned with server normalization.

## Supported Wallets

| Type | Standard | Description |
|------|----------|-------------|
| EOA | EIP-191 | Standard `personal_sign` (MetaMask, Coinbase, Rainbow) |
| Smart Contract | EIP-1271 | Contract wallets (Safe, Argent, Sequence) |
| Counterfactual | EIP-6492 | Pre-deployed smart accounts not yet on-chain |
