One **proof receipt ID** (`proofId`, same value as `qHash` on the wire) feeds gates, checks, and shareable proof pages. **[Proof receipts](./platform/receipts-and-results)** defines what to save. **[Security and trust](./platform/security-and-trust)** defines visibility.

## 1. Create your account

1. Sign in at [`https://neus.network/hub`](https://neus.network/hub) with passkey or wallet.
2. Complete profile basics so you have a verified wallet and can open **Identity**, **Apps**, and **Credits** when you need them.

## 2. Run your first loop locally

Use **[Quickstart](./quickstart)** for a direct `ownership-basic` proof with the SDK, or **[Hosted Verify](./cookbook/auth-hosted-verify)** when the flow is interactive (social, org, human). **[Integration](./integration)** describes check, verify, save, reuse for every path.

## 3. Pick how you integrate

**[Choose an integration path](./choose-an-integration-path)** matches outcomes to surfaces: hosted verify, SDK, VerifyGate, `gateCheck`, MCP.

## 4. Platform setup for production domains

When you need DNS domain proof, org OAuth, app linking, or sponsor grants at scale, follow **[Platform setup](./platform/setup)** after you understand the integration loop.

## Before production

Complete this block before production traffic. Details live in **[Billing](./platform/billing)** and **[Platform setup](./platform/setup)**.

### Register the integration and capture `appId`

1. Open **[Apps](https://neus.network/profile?tab=apps)**.
2. **Link app** or **Manage app**.
3. Enter your **app domain** (for example `acme.com`), set a **daily credit cap**, and finish the modal.

When registration succeeds, NEUS issues an **`appId`**. Send it as **`X-Neus-App`** on API traffic (the SDK maps this automatically).

```javascript
import { NeusClient } from '@neus/sdk';

const client = new NeusClient({
  appId: 'your-app-id', // same value sent as X-Neus-App
});
```

<Tip>
  **`appId`** is an attribution identifier (which product made the request). It does **not** by itself move charges to your organization wallet. Use sponsor grants when you want to pay on behalf of users.
</Tip>

### Credits and who pays

1. **Credits** reflect balance on the **payer wallet**.
2. If users self-pay, they need credits on **their** wallet. If your product pays, fund the org wallet and use **sponsor grants**. See **[Billing](./platform/billing)**.

### Sponsor grants (`sponsorGrant` when you pay for users)

If your product should pay verification costs for end users, fund the org wallet, issue short-lived sponsor grants, and pass them to the SDK as **`sponsorGrant`** (maps to **`X-Sponsor-Grant`**).

If users should self-pay, do not send sponsor grants.

```javascript
const client = new NeusClient({
  appId: 'your-app-id',
  sponsorGrant: '<sponsor-grant-token>',
});
```

### Access keys

When you need a **personal API key** (same **`Authorization: Bearer`** token for **`api.neus.network`** and for MCP when you want account-scoped tools), open **[Access keys](https://neus.network/profile?tab=account)** and generate one. You can skip this until you run scripts, server calls, or MCP flows that return **`auth_required`**.

### Production readiness checklist

- [ ] Integration domain is registered with a realistic daily cap
- [ ] Paying wallet has credits (yours or users', based on your model)
- [ ] Client sends **`appId`** and uses **`sponsorGrant`** only when sponsoring
- [ ] Team understands **[Billing](./platform/billing)** and **[x402](./platform/x402)** behavior
- [ ] Verification receipts are stored and reused in your product logic (see **[Proof receipts](./platform/receipts-and-results)**)
