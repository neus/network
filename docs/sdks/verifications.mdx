Use these patterns to choose how your app creates or reuses proofs. In most products, `reuse-or-create` is the default.

## Privacy and Visibility

| Option | When to Use |
|--------|-------------|
| `privacyLevel: 'private'` | Vaulted proofs for signed-in owner or authorized-session access only. |
| `privacyLevel: 'public'` | Reusable proofs. Pair with `publicDisplay` to choose public vs unlisted. |
| `publicDisplay: false` | Unlisted: usable in checks and direct lookups, hidden from discovery. |
| `publicDisplay: true` | Show proof on public profile (requires `public` privacy). |

Default to `{ privacyLevel: 'public', publicDisplay: false }` for most verify and gate flows. Use `private` for vaulted content and authorized reads only.

## Reuse-First Flow

Use this as the default UX. It minimizes user prompts while still issuing new proofs when needed.

```jsx
<VerifyGate
  requiredVerifiers={['ownership-basic']}
  strategy="reuse-or-create"
>
  <Protected />
</VerifyGate>
```

## Fresh Proof Flow

Use this when the action is high stakes and you need a newly created assertion.

```jsx
<VerifyGate
  requiredVerifiers={['wallet-risk']}
  strategy="fresh"
>
  <Protected />
</VerifyGate>
```

## Hosted Interactive Flow

Interactive verifiers hand off to NEUS hosted verification (`/verify`).

```jsx
<VerifyGate
  requiredVerifiers={['ownership-social']}
  hostedCheckoutUrl="https://neus.network/verify"
/>
```

## Policy checks from your app

Use the SDK **`gateCheck`** helper instead of downloading full proof payloads for every decision.

```javascript
const result = await client.gateCheck({
  address: '0x...',
  verifierIds: ['token-holding'],
  contractAddress: '0x...',
  minBalance: '100',
  chainId: 8453, // Asset chain where the token lives — required for asset verifiers only
  since: Date.now() - 60 * 60 * 1000
});
```
