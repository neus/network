Use this flow when you need to identify an agent, define what it can do, and let an app or AI runtime consume that result.

## What this flow includes

1. Agent registration
2. Optional agent permissions
3. A runtime such as MCP or the API that consumes the result

## Recommended Sequence

<Steps>
  <Step title="Register the agent">
    Create an agent registration proof that binds the agent wallet and its public metadata to one reusable proof receipt.
  </Step>
  <Step title="Add agent permissions when a controller is involved">
    If a person, team, or service is granting authority to the agent, create an agent permissions proof with scope, permissions, and any limits.
  </Step>
  <Step title="Use MCP or the API in the runtime">
    Your app, agent runtime, or tool can read the verifier catalog, check access, and send users to hosted verify when interactive completion is still needed.
  </Step>
</Steps>

## What you configure

- The agent wallet
- The agent label, type, and public description
- Optional permissions, scope, expiry, and limits when a controller is delegating authority

## What your app gets back

- A proof receipt ID for agent registration
- An optional proof receipt ID for agent permissions
- A stable way to reuse those results in hosted verify, MCP, and API checks

## Hosted handoff

When `neus_agent_link` (MCP) returns `status: "link_required"`, the user is sent to hosted verify to complete agent identity and delegation. The URL can include:

| Param | Use |
|-------|-----|
| `preset=agent-pack` | Preload agent-identity, agent-delegation, wallet-risk |
| `agent=<wallet>` | Pre-fill agent wallet when linking |

Example: `https://neus.network/verify?preset=agent-pack&agent=0x...`
