Most production integrations should follow one loop: check first, verify only when needed, save the proof receipt ID, then reuse it everywhere downstream.

## Production Checklist

<Steps>
  <Step title="Start with the lightest surface">
    Use hosted verify, widgets, SDK, API, or MCP based on the product job.
  </Step>
  <Step title="Check eligibility before creating a new proof">
    Use gate checks or reuse-first behavior so returning users are not prompted again without reason.
  </Step>
  <Step title="Use hosted verify for interactive work">
    Social accounts, team checks, and human verification should resolve through `/verify`.
  </Step>
  <Step title="Save the proof receipt ID">
    Persist the returned proof receipt ID with the record that needs it later.
  </Step>
  <Step title="Reuse the same proof receipt">
    Use it for status, display, proof pages, access checks, and policy instead of rebuilding trust logic per feature.
  </Step>
</Steps>

## Choose a Verification Strategy

| Mode | Best for | Result |
|------|----------|--------|
| `reuse-or-create` | Standard product UX | Reuse an eligible proof or create one if missing |
| `fresh` | High-stakes actions | Force a new proof |
| `reuse` | Read-only checks | Never create new proofs |

## Default Public Paths

| Need | Recommended path |
|---|---|
| Guided user verification | `https://neus.network/verify` |
| Browser or server integration | `@neus/sdk` |
| Server-side checks | `GET /api/v1/proofs/check` |
| Proof page display | `https://neus.network/proof/[proof-receipt-id]` |
| Agent tooling | `https://mcp.neus.network/mcp` |

## Go-Live Rules

<Checklist>
  <Item>Use `/verify` as the single hosted path for interactive flows.</Item>
  <Item>Read verifier metadata from the live catalog.</Item>
  <Item>Persist proof receipt IDs, not raw signing material.</Item>
  <Item>Use freshness windows for point-in-time checks.</Item>
  <Item>Test the full loop: create proof, save ID, open proof page, reuse proof for access.</Item>
</Checklist>
