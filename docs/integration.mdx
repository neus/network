Every NEUS surface follows one flow: **check** for an existing valid proof, **verify** only when needed, **save** the proof id, **reuse** it for gates and decisions.

**Check** for a valid result before a new verify. **Save** the **proof ID** from the response. API field: **`proofId`**.

## Where your code runs (default)

| In the browser | On your server |
|----------------|----------------|
| `NeusClient` verify, polling, widgets, **Hosted Verify** | **`gateCheck`**, allow/deny, secrets, payment or app-only headers, raw HTTP |
| Same-origin to your app only | Call `https://api.neus.network` from Node or your API route |

Use the **SDK** (or hosted links) in the browser so NEUS sees supported headers and paths. Do **not** hand-roll cross-origin `fetch` to the API from frontend code with extra headers; CORS allowlists differ by route and tooling will mis-implement it. Anything that needs headers the SDK does not set in the browser belongs behind **your backend** (proxy). See [JavaScript SDK](./sdks/javascript) and [API overview](./api/overview).

## Loop

1. **Check** - saved id or proofs/check
2. **Verify** - only if needed (hosted, SDK, or API)
3. **Save** - proof ID in your app (per user or device)
4. **Reuse** - UI, VerifyGate, server, MCP

| Situation | Approach |
|-----------|----------|
| Repeat visits | Run an eligibility check or use a stored proof ID before a new verify |
| Balance / NFT gates | Narrow `since` or `fresh` strategy |
| Org and domain | Link the app, then use the same loop |

## Visibility

| Mode | Use |
|------|-----|
| Private | Default for most products |
| Unlisted public | Public reuse without listing |
| Listed public | Discoverable attribution |

SDK `client.verify()` defaults **private**. `VerifyGate` create also defaults **private**.

## Strategies (SDK / API)

| Strategy | When |
|----------|------|
| `reuse-or-create` | Default: reuse if valid, else verify |
| `fresh` | Always new verification |
| `reuse` | Existing verification only |
