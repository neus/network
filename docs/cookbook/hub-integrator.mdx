Use this guide when you are connecting a website, community, app, team, or partner flow to NEUS. After **[Get started](../get-started)**, you will use hosted verify, save proof receipt IDs, and reuse them across profile trust, gating, and attribution.

## What NEUS gives you

| Capability | What it solves |
|---|---|
| Portable profile trust | A person can link wallets, socials, domains, team association, and proof-backed badges over time |
| Domain and namespace trust | Sites, communities, and orgs can prove control of a domain and use that trust in onboarding and identity flows |
| Linked apps | Apps and platforms can attribute usage and connect the app to the same proof layer |
| Proof receipts | Every completed flow returns a reusable proof receipt ID instead of a one-time verification dead end |
| Hosted verify | Interactive checks stay on the hosted `/verify` flow instead of forcing partners to build custom UX |

## Prerequisites

- A NEUS account with a verified wallet
- Your public app or site domain

## 1. Domain Verification

Link your domain to your wallet by setting a DNS TXT record. This is the foundation for site trust, partner onboarding, app linking, and namespaced identity claims.

**Step 1:** Add a DNS TXT record at `_neus.<yourdomain.com>`:

```text
neus=<your-wallet-address>
```

**Step 2:** Create a domain proof:

```javascript
const proof = await client.verify({
  verifier: 'ownership-dns-txt',
  data: {
    domain: 'yourdomain.com',
    walletAddress: '0x...',
  },
  walletAddress: '0x...',
  signature: '0x...',
  signedTimestamp: Date.now(),
});
```

**Step 3:** Confirm the proof is active:

```bash
GET /api/v1/proofs/check?verifierIds=ownership-dns-txt&address=0x...
```

## 2. Team and Organization Verification

When a site, partner, or company needs to verify that someone belongs to a workspace, send them to hosted verify and let NEUS handle the guided Google or Microsoft flow.

### Setup Steps

1. Launch `https://neus.network/verify`
2. Use `preset=org-pack` when organization membership is the only requirement
3. Add `returnUrl` or popup mode for completion handling
4. Read the returned proof receipt ID after the hosted flow finishes

### Recommended Hosted URL

```text
https://neus.network/verify?preset=org-pack&returnUrl=https://app.example.com/verified
```

Popup example:

```text
https://neus.network/verify?preset=org-pack&mode=popup&origin=https://app.example.com
```

### Restrict to One Company Domain Only When You Need It

If access should only succeed for members of one company or team domain, pass `expectedOrgDomain`.

```jsx
<VerifyGate
  hostedCheckoutUrl="https://neus.network/verify"
  requiredVerifiers={['ownership-org-oauth']}
  verifierData={{
    'ownership-org-oauth': {
      provider: 'google',
      expectedOrgDomain: 'acme.com',
    },
  }}
>
  <ProtectedContent />
</VerifyGate>
```

<Note>
You typically do not need your own Google or Microsoft OAuth app to use the hosted NEUS flow.
</Note>

## 3. Link the app or site on the platform

Link your integration domain on the **NEUS platform** to receive attribution, usage tracking, and credit controls:

| Step | Action |
|------|--------|
| 1 | Go to the [NEUS platform](https://neus.network/hub) → Profile → Apps |
| 2 | Enter app domain, for example `app.example.com` or `https://app.example.com` |
| 3 | Set daily credit cap |
| 4 | Use `*` for any origin only in development |
| 5 | Send `X-Neus-App` and `Origin` headers from your approved domain |

## 4. App Attribution (API)

To track usage per integration through the same proof layer:

1. Create or obtain an `appId` for your integration
2. Treat `appId` as a public attribution identifier, not a secret
3. Include `X-Neus-App: <appId>` on all API requests

```javascript
const client = new NeusClient({ appId: 'your-app-id' });
```

## 5. Combine trust signals

You can combine these capabilities into one portable verified profile:

- linked apps
- domain verification
- team and organization verification
- username and namespace claims
- linked wallets
- public, private, and unlisted proof visibility
- proof pages and your platform profile

On the platform, the most common flows are:

| Flow | Where | Notes |
|------|-------|------|
| Domain | Profile -> Identity -> Domains | Add DNS TXT at `_neus.<domain>` first |
| Organization | Profile -> Identity -> Organizations | Hosted NEUS verify for Google Workspace or Microsoft 365 |
| Username | Profile / Claim flows | Portable handle and namespace claims |
| Linked wallets | Profile identity flows | Multiple wallets as one linked identity |
| Privacy | Per proof | Public, private, or unlisted proof visibility |

Link the app first in Profile -> Apps, then connect whichever trust signals your product actually needs.

## 6. Hosted Verify for Interactive Flows

Interactive checks such as linked accounts, team verification, and human verification should resolve through hosted verify.

### React (VerifyGate)

```jsx
import { VerifyGate } from '@neus/sdk/widgets';

<VerifyGate
  apiUrl="https://api.neus.network"
  hostedCheckoutUrl="https://neus.network/verify"
  requiredVerifiers={['ownership-org-oauth']}
  verifierData={{
    'ownership-org-oauth': {
      provider: 'google',
      expectedOrgDomain: 'acme.com',
    },
  }}
  onVerified={(result) => console.log('Verified:', result.proofId)}
>
  <ProtectedContent />
</VerifyGate>
```

If you do not need to restrict a specific company domain, prefer the simpler preset URL and avoid custom payloads.

## 7. Sponsor Grants

If your integration sponsors verification costs:

```javascript
const client = new NeusClient({
  appId: 'your-app-id',
  sponsorGrant: '<sponsor-grant-token>',
});
```

<Warning>
Sponsor grant tokens are short-lived. Re-issue them per session. Do not embed long-lived API keys in client-side code.
</Warning>

## 8. Usage and Analytics

Track usage with your `appId` and your own reporting:

- proof creation volume by capability
- gate check counts by app
- sponsor grant consumption

The `/api/v1/proofs/check` response includes `matchCount` for telemetry.

## Portability Model

NEUS is most useful when you follow this model:

- a person has a reusable verified profile
- that profile can accumulate wallets, linked accounts, domains, team association, and proof-backed badges over time
- apps can attribute usage and gate access using the same proof layer
- domains and namespaces can anchor site, community, and organization identity
- proof receipts are the portable artifact shared across all of it

## Integration guidelines

- Keep `/verify` as the single hosted path for interactive verification
- Route OAuth and interactive verifier handoffs through NEUS hosted verification
- Do not create parallel auth routes when NEUS already provides a standard path
