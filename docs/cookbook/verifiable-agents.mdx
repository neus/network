## Agent Identity

Use `agent-identity` to register an AI or automation agent:

```javascript
await client.verify({
  verifier: 'agent-identity',
  data: {
    agentId: 'my-agent',
    agentWallet: '0x...',
    agentType: 'ai',
    description: 'My AI assistant',
    capabilities: ['web-search'],
    instructions: 'System prompt (max 4000 chars)',
    skills: ['web-search', 'code-execution'],
    services: [
      { name: 'metrics', endpoint: 'https://metrics.example.com/v1', version: '1.0' }
    ]
  },
  walletAddress: controllerWallet,
});
```

### Fields

| Field | Required | Description |
|-------|----------|-------------|
| `agentId` | Yes | Unique agent identifier |
| `agentWallet` or `agentAccountId` | One required | Agent's wallet (EVM/Solana) or CAIP-10 account |
| `agentType` | No | `ai`, `bot`, `service`, `automation`, `agent` |
| `description` | No | Agent description |
| `capabilities` | No | List of capabilities |
| `instructions` | No | System prompt (max 4000 chars) |
| `skills` | No | List of skills (max 48) |
| `services` | No | Service endpoints (max 16) |

## Agent Delegation

Use `agent-delegation` to grant permissions to an agent:

```javascript
await client.verify({
  verifier: 'agent-delegation',
  data: {
    controllerWallet: '0x...',  // Must match signer
    agentWallet: '0x...',
    scope: 'payments:x402',
    permissions: ['execute', 'read'],
    maxSpend: '1000000000000000000',  // 1 ETH in wei
    allowedPaymentTypes: ['x402'],
    receiptDisclosure: 'summary',
    expiresAt: Date.now() + 7 * 24 * 60 * 60 * 1000, // 7 days
    instructions: 'Policy instructions for the agent',
    skills: ['market-data']
  },
  walletAddress: controllerWallet,
});
```

### Delegation Fields

| Field | Required | Description |
|-------|----------|-------------|
| `controllerWallet` | Yes | Must match signer |
| `agentWallet` | Yes | Agent's wallet |
| `scope` | No | Permission scope |
| `permissions` | No | `execute`, `read`, etc. |
| `maxSpend` | No | Maximum spend in wei |
| `expiresAt` | No | Expiration timestamp |

## Agent Identity + Linked Proofs (OpenClaw / MCP)

When you use an agent identity wallet with linked proofs (e.g., as your OpenClaw agent identity):

1. **Wallet is the anchor.** Your agent-identity proof attests that a wallet controls the agent. All other proofs (ownership, social, domain, etc.) are linked to that same wallet.

2. **Resolve by wallet.** Consumers (OpenClaw, MCP, gate checks) resolve proof context via `GET /api/v1/proofs/byWallet/{address}`. No need to embed proof IDs in the agent-identity proof—the wallet is the lookup key.

3. **Reference pattern (optional).** To explicitly reference another proof in content, use `ownership-basic` with `reference: { type: 'qhash', id: '<proofId>' }`. The raw reference type remains `qhash` for compatibility, but the value is still the proof receipt ID. For agent-identity, `skills` holds semantic identifiers (e.g., `openclaw-agent`); proof IDs are resolved by wallet, not embedded.

<Note>
**Best practice:** Use the agent wallet as the single identity. Fetch all proofs for that wallet when building agent context. The agent-identity proof attests control; linked proofs provide additional trust signals.
</Note>
