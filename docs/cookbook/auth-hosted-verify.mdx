Use hosted verify when users need to sign, link OAuth, or complete interactive verifiers. NEUS provides a hosted UI for these flows.

NEUS supports a signature-based protocol for proofs, and an optional session layer for first-party browser UX.

## Two Layers

| Layer | Purpose |
|-------|---------|
| **Protocol signing (proofs)** | What verifiers require; described in [Signing](../api/authentication) |
| **Session (`/api/v1/auth/*`)** | Optional convenience layer to reduce repeated prompts |

## Hosted Verify UX (Passkey or Wallet)

When a flow requires human interaction (signature, OAuth linking, proof creation), NEUS provides a hosted UI.

- Users can **sign in with a passkey** (no wallet extension required) **or connect a wallet**
- After completion, the hosted UI returns the user to your app

After successful verification, users can be redirected to the [Hub](https://neus.network/hub) or [Profile](https://neus.network/profile) to manage identity, link apps, or view proofs.

### Hosted URL

**Base:** `https://neus.network/verify`

| Param | Use |
|-------|-----|
| `intent=login` | Session-first login flow |
| `verifiers=<ids>` | Comma-separated verifier IDs (e.g. `verifiers=wallet-risk`) |
| `preset=<id>` | Preset bundle (e.g. `preset=agent-pack`). See [Presets](../platform/presets) for full catalog. |
| `gateId=<id>` | Gate requirements (verifiers derived from gate) |
| `returnUrl=<url>` | Redirect target after completion |
| `mode=popup&origin=<url>` | Popup flow; `postMessage` to opener |

<Warning>
Do not invent a second hosted login route. `intent=login` on `/verify` is the correct implementation.
</Warning>

## Session SSOT: `GET /api/v1/auth/me`

If you use the session layer, treat this endpoint as the **single source of truth** for:

- Whether the user is authenticated
- The user's identity pack (subjectId/did/address)
- Onboarding readiness (if available for your integration)

### Routing Contract

| State | Action |
|-------|--------|
| `authenticated: false` | Keep hosted login choices visible |
| `authenticated: true` + `needsOnboarding: true` | Your app decides: prompt profile setup or proceed |
| `authenticated: true` + `needsOnboarding: false` | Route to `returnUrl` or your app's authenticated landing |

## Hosted Checkout URL Configuration

If you run the NEUS API at a custom `apiUrl`, also set the hosted verify URL explicitly:

```jsx
<VerifyGate
  apiUrl="https://api.neus.network"
  hostedCheckoutUrl="https://neus.network/verify"
  requiredVerifiers={["ownership-basic"]}
>
  <Protected />
</VerifyGate>
```

## Popup Completion Signaling

For popup flows, the hosted window posts a completion message to the opener.

| Message type | When |
|--------------|------|
| `neus_login_complete` | After session/login is confirmed |
| `neus_checkout_done` | After verification completes (includes the proof receipt ID, `proofId`, `verifierIds`, and optional `gateId`) |

**Delivery:** `window.opener.postMessage(...)`. Origin is restricted to the supplied `origin` when it is an absolute HTTP(S) URL.

## Outage Contract

The hosted verify layer normalizes upstream outages:

- Interactive auth routes return `503` for upstream `5xx` or transport timeouts
- They include `Retry-After: 10`
- They include `X-Neus-Upstream-Status` for diagnostics
- They set `Cache-Control: no-store`
