Start with [Get started](../get-started) for the setup checklist. This page covers billing details, edge cases, and x402.

## Why this can feel confusing

`appId` on `NeusClient` sends `X-Neus-App` so NEUS can attribute usage to your product. It does not, by itself, move the bill to your NEUS account balance.

Credits are charged to a payer wallet resolved from how the request is authenticated: the signing wallet, a sponsor grant, an x402 receipt, or certain gate setups. Until you add one of those, `appId` alone does not move billing.

## Apps vs individuals

| You are... | Priority | Billing |
|----------|------------------------|------------------------|
| Building an app | Predictable cost and no random out-of-credits walls | You pay for metered verification and checks via sponsor grants and credits on your org wallet |
| Using NEUS for yourself | Your own proofs and profile | You pay with your NEUS credits, or x402 when you run out |

Same SDK and API - different payer context.

## Recommended path for app integrators

1. Create and configure your app.
2. Keep credits on the organization wallet that should pay.
3. Issue short-lived `X-Sponsor-Grant` tokens so billable calls debit your org balance, not the end user's.
4. Use `new NeusClient({ appId: '...', sponsorGrant: token })` so every request is labeled and paid under your grant.

<Warning>
Do not rely on anonymous traffic for heavy gate checks or verification volume. Without a payer context, behavior and limits are not the same as production sponsored traffic.
</Warning>

## Recommended path for individuals

1. Use Hosted Verify or SDK with your own wallet.
2. Credits come from your NEUS balance on that wallet.
3. If you hit `402`, use x402 and retry with `PAYMENT-SIGNATURE`.

## How NEUS selects the payer

Precedence: paid receipt -> sponsor grant -> gate owner pays if configured -> wallet that signed the request -> x402 paywall path -> otherwise you may hit limits or payment required.

You do not send "bill this appId" in the body and expect the wallet to switch. Payer is derived from crypto and session mechanics, not from `appId` alone.

## SDK configuration

| Config | Effect |
|--------|--------|
| `appId` | Sets `X-Neus-App` for analytics and attribution |
| `sponsorGrant` | Sets `X-Sponsor-Grant` so your org pays when the grant is valid |
| `paymentSignature` | Sets `PAYMENT-SIGNATURE` for x402 retries |

The SDK does not auto-complete x402 payments. Your app handles 402 -> pay -> retry, or you avoid 402 by keeping org credits healthy.

## Visibility vs billing

- Public and unlisted proofs can be checked without the user signing in.
- Private visibility rules still apply; billing is separate.
