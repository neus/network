**Verifier ID:** `ownership-basic`

## Best for

- Content and file provenance
- Authorship and licensing claims
- Portable proof IDs (listings, bounties, submissions)

## Payload

| Field | Required | Notes |
|------|----------|-------|
| `owner` | Yes | Must match signer |
| `content` | No | Text body to prove and optionally store with the proof. Use this for plain text, release notes, claims, descriptions, or other human-readable content. When stored and visible, NEUS exposes it on the proof as `proof.publicContent.content`. Max 50k chars. |
| `contentHash` | No | Pre-hashed body; verifier can integrity-check against `content` if both sent |
| `reference` | No | Pointer (URL, IPFS, tx, file id, …). Server does not fetch it. Send `content` if you need bytes stored. |
| `provenance` | No | Human / AI / mixed (advisory) |

## Stored proof content

Send `content` when you want NEUS to store a readable text body with the proof.

NEUS keeps `content` as the ownership-basic verifier input and maps stored readable content onto the normalized proof response as `proof.publicContent.content` when visibility allows it.

Use `contentHash` or `reference` when you only want to prove a hash or external pointer without storing the text body.

For visibility defaults and when stored text appears on public proofs, see [Security and trust](../platform/security-and-trust).

## Example

```json
{
  "owner": "0x...",
  "content": "Release notes for v1.2",
  "reference": {
    "type": "url",
    "id": "https://example.com/releases/v1-2"
  }
}
```

## Retrieval

When reading proofs back, stored text appears on the normalized proof as `proof.publicContent.content` when visibility allows it; otherwise rely on a stable `reference`. Responses may include `contentResolutionStatus`: `resolved` or `hash_only_not_recoverable`.
