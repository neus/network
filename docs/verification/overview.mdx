Each verification returns a portable proof receipt ID that can be reused across products.

## Default Path

1. **Choose the required proof or policy outcome**
2. **Run Hosted Verify, SDK, or API verification**
3. **Store the returned proof receipt ID**
4. **Reuse the receipt for access, display, onboarding, or policy**

## What NEUS Verification Powers

| Outcome | Typical use today |
|---|---|
| Identity and association | Linked accounts, domains, usernames, team verification, proof-backed profiles |
| Ownership and provenance | Content authorship, file receipts, submissions, project operator checks, source context |
| Access and monetization | Premium access, protected communities, gated routes, high-trust unlock flows |
| Abuse reduction | Human verification and wallet safety screening before sensitive actions |
| Agent operations | Agent registration, delegated permissions, hosted handoff, and MCP workflows |

## Key Terms

| Term | Meaning |
|------|---------|
| Hosted verify | The standard `/verify` flow for guided and interactive verification |
| Request | The capability set plus the normalized proof payload |
| Signing string | The deterministic wallet message returned by `POST /api/v1/verification/standardize` |
| Proof receipt ID | The stable identifier returned for a proof |
| API path alias (`qHash`) | Raw HTTP path name for the same proof receipt ID |

## Reuse Rules

- Save the proof receipt ID once.
- Reuse it for proof pages, profile trust, eligibility checks, and policy decisions.
- Send users back through hosted verify only when you need an interactive flow or a fresh point-in-time result.
- Prefer checking saved receipts and eligibility over rebuilding trust logic in each product.
