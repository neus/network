Personal access keys: [Access keys](https://neus.network/profile?tab=account). Same **`Authorization: Bearer`** pattern as [API authentication](../api/authentication). Apply on the MCP HTTP connection ([Setup](./setup)).

| Auth | What works |
|------|------------|
| **None** | `neus_context`, `neus_proofs_check`, public-scope `neus_proofs_get`, verifier catalog reads ([Overview](./overview)) |

| Bearer (`npk_*`) | Detail |
|------------------|--------|
| Unlocks | `neus_me`, authenticated `neus_proofs_get`, profile reads |
| `neus_verify` (instant) | No extra signature when **profile wallet** matches **`walletAddress`** ([Overview](./overview)) |
| Interactive verifiers | [Hosted Verify](../cookbook/auth-hosted-verify) |

#### Header

```
Authorization: Bearer <npk_...>
```

Use **`npk_*`** when a tool needs **your account** (e.g. `neus_me`, private reads)—not the end user’s browser login.

**With Bearer:** call `neus_context`, then `neus_me` → expect **`status: ok`**.

| Situation | What to do |
|-----------|------------|
| User needs hosted passkey, OAuth, or social steps | Open **`neus.network/verify`** using **`hostedVerifyUrl`** from the tool response. |
| User has a connected wallet | **`neus_verify`** with signature, or the hosted URL when the tool returns it. |
| IDE or server automation | **`npk_*` Bearer** on MCP; direct HTTP uses the same header ([API authentication](../api/authentication)). |

| Topic | Rule |
|-------|------|
| Store keys | Server or IDE MCP config only. |
| Rotate | Re-issue from Access keys if exposed. |
| `hostedVerifyUrl` | Send the user to **`neus.network/verify`**. |
