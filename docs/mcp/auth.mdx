Use a **personal access key** from [Access Keys](https://neus.network/profile?tab=account) when your MCP client needs access to your NEUS account or private data.

NEUS MCP uses the same **`Authorization: Bearer`** pattern as [API authentication](../api/authentication). Apply the header on the MCP HTTP connection during [setup](./setup).

## Browser sign-in (recommended)

```bash
npx -y -p @neus/sdk neus auth
```

The CLI opens your browser to a hosted sign-in page. After you authenticate (passkey or wallet), the CLI receives an attestation token and configures your MCP clients automatically — no manual key copy-paste needed.

Use browser sign-in for personal setups. The `--access-key` variant below is for servers and automation where a browser is not available.

## Quick answer

| Mode | Use it for |
|------|------------|
| **No authentication** | `neus_context`, `neus_proofs_check`, public `neus_proofs_get`, and verifier catalog reads |
| **Bearer (`npk_*`)** | `neus_me`, authenticated `neus_proofs_get`, profile reads, and account-aware MCP flows |

## Authorization header

```http
Authorization: Bearer <npk_...>
```

Use **`npk_*`** when a tool needs **your account**. Do not use an end user's browser session for MCP authentication.

Authenticated MCP sessions should reuse receipts before any browser step: **`neus_context`** → **`neus_me`** → **`neus_agent_link`** (when an agent wallet is in play) → **`neus_proofs_check`** → **`neus_proofs_get`** when you need proof details. Use **`neus_verify_or_guide`** only when something required is still missing.

For **agent wallets**, follow [Agent verification flow](../agents/agent-verification-flow): call **`neus_agent_link`** first when that address already exists; use **`neus_agent_create`** only when the link response shows gaps.

## What Bearer unlocks

* `neus_me`
* authenticated `neus_proofs_get`
* profile reads
* account-aware MCP flows

`neus_verify` can complete without an additional signature when your authenticated profile wallet already matches the request **`walletAddress`**. See [Overview](./overview).

## Interactive verification flows

[Hosted Verify](../cookbook/auth-hosted-verify): passkey, wallet, OAuth, and social **proof** steps run **on NEUS**. If a tool returns **`hostedVerifyUrl`**, open it once, then continue in MCP.

## Common situations

| Situation | What to do |
| --- | --- |
| User needs passkey, OAuth, or social verification | Open the returned **`hostedVerifyUrl`**. |
| User has a connected wallet | Use **`neus_verify`** with a signature, or the hosted URL when the tool returns one. |
| IDE or server automation | Set **`Authorization: Bearer <npk_...>`** on MCP. Raw HTTP uses the same pattern: [API authentication](../api/authentication). |

## Security

| Topic | Rule |
| --- | --- |
| Store keys | Server or IDE MCP configuration only. |
| Browser exposure | Never expose personal access keys in browser code. |
| Rotation | Re-issue the key from [Access Keys](https://neus.network/profile?tab=account) if it is exposed. |
| `hostedVerifyUrl` | Send the user to the returned NEUS hosted flow. |
